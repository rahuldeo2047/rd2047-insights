Act 1  ·  The Concept

What is Visual-HR-Est?

Multi-face contactless heart rate estimation from a single RGB camera. Remote photoplethysmography (rPPG) / Eulerian magnification. The camera detects subtle skin-color changes caused by blood pulsing through facial vessels — imperceptible to the eye, measurable in the pixel-level signal.


Magic-mirror form factor: semi-transparent mirror glass over a display. Camera sits behind the mirror. The subject sees their reflection and a BPM overlay simultaneously. No wearable, no contact. Target accuracy: ±2 BPM MAE — medical-grade devices are typically ±3 BPM.

Why ±2 BPM?
Medical-grade pulse oximeters (FDA Class II) specify ±2–3 BPM. Best published rPPG systems (CHROM, POS) achieve 2–5 BPM MAE in controlled lighting with one subject. Multi-face simultaneous measurement in uncontrolled lighting pushes the error higher. ±2 BPM is the target, not the expected outcome. If ±5 BPM is achieved, that is still useful and honest.
Why build this after WiFi-HR-Est?
WiFi-HR-Est is the privacy-first path — RF sensing, no image data. Visual-HR-Est is the higher-accuracy path — rPPG has better published validation than CSI-HR. The two systems cross-validate each other. Operator decision: build Visual-HR-Est after WiFi-HR validation gap is closed. Starting from scaffold now.

Act 2  ·  Privacy Posture

Privacy — non-negotiable constraints

A camera-based biometric system has one legitimate path to deployment: explicit consent, minimal data footprint, and full local processing. These are not features — they are requirements. Any implementation that does not satisfy all five constraints does not ship.

🚫
No frame egress
No image or video data leaves the device. All processing is on-device (RPi5 + Hailo). No cloud inference, no remote logging of frames.
✅
Opt-in only
Visible LED indicator when camera is active. System does not operate without explicit opt-in. Indicator is hardware-level, not software-controlled.
🔢
Anonymous IDs only
Subjects identified by unsigned integer session IDs only. No name, no face embedding stored, no identity linkage. ID resets each session.
🌫️
Auto-blur option
Optional: faces blurred in any local debug display. ROI coordinates extracted but raw frames discarded after ROI extraction.
📋
On-device audit log
Local log of session start/end timestamps and subject count (no BPM values logged by default). Operator-readable only.

Hailo-8L rationale
RetinaFace or YuNet face detection at 30fps is feasible on RPi5 CPU. The Hailo-8L (13 TOPS) accelerates it and frees CPU for rPPG signal processing. The Hailo-10H (40 TOPS) on the edge-AI rig is the proven platform — the 8L is the more cost-appropriate choice for a dedicated mirror device. Decision deferred until after WiFi-HR validation.
Act 4  ·  Algorithm Roadmap

Planned algorithm evaluation order

Four algorithms in evaluation order. Classical methods first — deep learning only if classical ceiling is met. Evaluate on single-face controlled before attempting multi-face.

EVALUATE FIRST
POS (Wang 2017)
Plane Orthogonal to Skin — current SOTA reference for classical rPPG. Handles illumination changes better than CHROM in theory. Scikit-bio / custom implementation.
EVALUATE SECOND
CHROM (de Haan 2013)
Chrominance-based. Robust to motion. Most-cited classical method. Published MAE: 2–4 BPM in controlled conditions. The comparison baseline.
EVALUATE THIRD
Eulerian Video Magnification
MIT Wu 2012. Temporal color magnification. Strong visual demo value. Not necessarily the most accurate. Useful for explaining the concept publicly.
ONLY IF NEEDED
Deep-rPPG
DeepPhys, PhysNet, RhythmNet. Deploy only if classical methods don't reach ±5 BPM MAE in multi-face scenario. Training data requirements are non-trivial.

Act 5  ·  Status

Scaffold only — build gate defined

This project is a design document. The build gate is explicit: rebuild begins after WiFi-HR-Est validation gap is closed (Bland-Altman study against PPG ground truth, N ≥ 10 subjects).

Build gate
Visual-HR-Est implementation starts when WiFi-HR-Est has a Bland-Altman result (even a disappointing one). The two systems are designed to cross-validate. Running both in parallel would make the cross-validation analysis ambiguous. WiFi-HR goes first, produces a measured MAE, then Visual-HR provides the comparison point.
